```java
 * Adds template_name, template_params_json, media_url columns to whatsapp_message_log
 * so retries can redeliver the original message content.

private void ensureWhatsAppMessageLogRetryColumns() {
    try {
        Integer count = jdbcTemplate.queryForObject(
            "SELECT COUNT(*) FROM information_schema.COLUMNS " +
            "WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = 'whatsapp_message_log' AND COLUMN_NAME = 'template_name'",
            Integer.class

                "ADD COLUMN template_params_json VARCHAR(1000) DEFAULT NULL, " +
                "ADD COLUMN media_url VARCHAR(500) DEFAULT NULL");

        }
    } catch (Exception e) {
        log.warn("DB migration skipped (whatsapp_message_log retry columns): {}", e.getMessage());
    }
```

## Task 3: Fix retry count off-by-one and scheduler missing RETRYING entries

**Files:**
- Modify: `backend/src/main/java/com/hms/service/whatsapp/WhatsAppService.java`
- Modify: `backend/src/main/java/com/hms/repository/WhatsAppMessageLogRepository.java`
- Modify: `backend/src/main/java/com/hms/scheduler/WhatsAppRetryScheduler.java`

**Context:**

Two bugs cause messages to get stuck forever:

**Bug A — `retry()` increments AFTER the `>= MAX_RETRIES` check.**
Current code:
```java
if (entry.getRetryCount() >= MAX_RETRIES) {   // e.g. 1 >= 2? NO
    entry.setStatus(STATUS_PERMANENTLY_FAILED);
} else {
    entry.setStatus(STATUS_RETRYING);
    entry.setRetryCount(entry.getRetryCount() + 1);  // retryCount becomes 2
    entry.setNextRetryAt(now + 15min);
}
```
After the 2nd retry fails: `retryCount` becomes 2 but the check `1 >= 2` was `false` so status is `RETRYING`. Next scheduler run: `retryCount < 2` → 2 is not < 2 → never picked up. Entry is stuck in RETRYING forever.

**Fix A:** Increment `retryCount` BEFORE the check.

**Bug B — Scheduler only queries `STATUS_FAILED`, never picks up `STATUS_RETRYING` rows.**
After a retry attempt fails, the status is set to `STATUS_RETRYING`. The scheduler query `status = 'FAILED'` never fetches these entries.

**Fix B:** Add a repository method that accepts a list of statuses, and use it in the scheduler.

- [ ] **Step 1: Fix `retry()` in `WhatsAppService.java`**

Find the failure branch in `retry()` (the `else` block after `if (success)`). Replace the entire `else` block:

```java
        entry.setErrorMessage("Max retries exceeded");

}
```

**Replace with:**

The only change is: `setRetryCount(+1)` moves to BEFORE the `if` check.

```java
/**
 * Adds missing indexes on frequently-queried columns.
 * Each index is created only if it does not already exist.
 * Adding indexes is safe — it never breaks existing queries.
 */
private void ensureMissingIndexes() {
    addIndexIfMissing("appointments", "idx_appt_date", "appointment_date");
    addIndexIfMissing("patients",     "idx_patient_hospital", "hospital_id");
    addIndexIfMissing("doctors",      "idx_doctor_hospital",  "hospital_id");
}

private void addIndexIfMissing(String table, String indexName, String column) {
    try {
        Integer count = jdbcTemplate.queryForObject(
            "SELECT COUNT(*) FROM information_schema.STATISTICS " +
            "WHERE TABLE_SCHEMA = DATABASE() AND TABLE_NAME = ? AND INDEX_NAME = ?",
            Integer.class, table, indexName
        );
        if (count != null && count == 0) {
            jdbcTemplate.execute(
                "ALTER TABLE `" + table + "` ADD INDEX `" + indexName + "` (`" + column + "`)"
            );
            log.info("DB migration applied: index {} added on {}.{}", indexName, table, column);
        }
    } catch (Exception e) {
        log.warn("DB migration skipped (index {} on {}): {}", indexName, table, e.getMessage());
    }
}
```
